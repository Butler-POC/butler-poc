## 0. 작업 환경에 대한 중요한 전제

작업 착수 시점에 저장소의 **모든 소스 파일이 빈 파일(0바이트)** 이었다. 즉 지시서가 "이미 존재하니 소비만 하라"고 한 기반(스캐폴딩, `llm.service`, `chat.socket`, `kakao.service`, `Building/Tenant/User` 모델·타입, `tokens.css`, 라우터, 공통 UI)이 실제로는 비어 있었다.

협의 결과 **"내 7개 기능만 작성하고, 빈 의존성의 인터페이스는 가정한다"** 방침으로 진행했다. 따라서:

- 본 슬라이스 코드는 **빈 기반 모듈을 import/참조**한다.
- **빌드·실행·런타임 검증은 기반 머지 후로 미뤄진다.** 현재는 코드 리뷰·타입 정합성 확인까지 가능.
- DB는 **AWS RDS MySQL 공용 DB**로 확정(지시서의 "SQLite" 문구는 무시. CLAUDE.md `provider=mysql` + 팀 통합 용이성 근거). `server/.env`의 `DATABASE_URL` 사용.

## 1. 도메인 모델 / 타입 (CP1)

새 도메인 **공실(Vacancy)** 과 **하자(Issue)** 를 도입했다. (Issue는 다음 하자 줄기에서 사용하지만 모델·타입은 함께 정의)

- `server/prisma/schema.prisma` — `Vacancy`·`Issue` 모델 + enum 추가.
  - **기반 비침투 결정**: `Building/Tenant/User`와의 관계를 정식 `@relation`으로 묶으면 양쪽 모델에 back-relation을 넣어야 해 기반 모델을 건드리게 된다. 이를 피하려고 **scalar FK(`buildingId`, `tenantId`)** 로만 참조했다. 정식 관계 연결은 머지 시점에 추가한다.
  - MySQL 특성 반영: 긴 텍스트 `@db.Text`, 사진 배열은 `String[]` 미지원이라 `photos Json?` 컬럼.
  - `datasource`·`generator` 블록은 기반(팀원) 소유라 본 파일에 포함하지 않았다. → 그래서 현재 `prisma validate` 는 단독 통과하지 않음(예정된 결과).
- `client/src/types/{vacancy,issue}.ts` ↔ `server/src/types/{vacancy,issue}.ts` — **동일 내용 복제**(shared 없음 규칙). `index.ts` 배럴에 슬라이스 추가분 export.
  - 양쪽 diff로 일치 확인 완료(자기참조 주석 한 줄 제외 동일).

**임의 설계한 값**(지시서 미지정): `IssueStatus = reported|reviewing|resolved`, `IssueCategory` 7종(plumbing/electrical/boiler/waterproof/interior/appliance/other), `VacancyStatus = available|closed`.

## 2. L-07 공실 등록 (오너)

- **서버**: `routes/vacancies.ts` + `controllers/vacancy.controller.ts`
  - `POST /api/vacancies` — 필수값(buildingId·unit·deposit·monthlyRent·areaM2) 검증 후 생성. `requireAuth` 보호.
  - `GET /api/vacancies?mine=true` — 로그인 임대인 소유 건물의 공실만. scalar FK 호환을 위해 **건물 ID 집합으로 필터**(소유 건물 조회 → `buildingId in [...]`).
- **클라**: `api/vacancies.ts`, `stores/vacancies.ts`(Pinia), `views/landlord/VacancyRegister.vue`
  - 한 화면에서 **등록 폼 + 내 공실 목록**을 함께 제공(IA 1화면).
  - 금액은 `₩` + `tabular-nums`, 빈 상태는 brass 점 + 한 줄("등록된 공실이 없습니다."), 합니다체.

## 3. A-01 공실 조회 (에이전트)

- **서버**: `GET /api/vacancies` (mine 아님) — "빈 방 + 임대인 정보"로 가공.
  - vacancies 조회 후 `buildingId` 집합으로 `prisma.building`(기반 모델, 머지 후 사용 가능)에서 `owner` 포함 조회해 `VacancyWithOwner`로 매핑. → 에이전트가 오너 연락 진입점 확보.
- **클라**: `views/agent/VacancyList.vue`
  - 매물 카드(PROPERTY CARD 규격) 리스트 + "모집중만 보기" 필터. 카드 → "임대인 연결" 버튼으로 A-02 진입.

## 4. A-02 건물주 연결 (에이전트 ↔ 오너)

- **공용 컴포넌트**: `components/domain/VacancyChat.vue` — 메시지 스레드 + 입력. 기반 `useChatStore`(소켓 플럼빙)를 소비.
- **에이전트측**: `views/agent/OwnerChat.vue` — 라우트 `vacancyId` + 본인 `agentId`로 방 키 생성.
- **오너측**: `views/landlord/VacancyChats.vue` — 내 공실 관련 에이전트 문의 수신함(스레드 목록 → 채팅).
- **채팅방 키 규약**: `vacancy:{vacancyId}:agent:{agentId}` (한 공실에 여러 에이전트가 각각 오너와 별도 대화). `api/vacancies.ts`의 `vacancyRoomId()`로 통일.

## 5. 가정한 기반 인터페이스 (머지 시 정합 필요)

| 대상 | 가정한 형태 |
|---|---|
| `server/src/lib/prisma` | PrismaClient 인스턴스 default export. 머지 후 `prisma.building` 사용 가능 |
| `Building` 모델 | `{ id, address, ownerId, owner: User }` — Vacancy와 scalar FK로 연결 |
| `server/src/middleware/auth` | `requireAuth` named export, `req.user = { id, role }` 주입 |
| `client/src/api/client` | axios 인스턴스 default export, baseURL `/api` |
| `client/src/stores/buildings` | `useBuildingsStore` → `{ items: {id,address}[], load() }` |
| `client/src/stores/auth` | `useAuthStore` → `{ user: {id, role, name} }` |
| `client/src/stores/chat` | `useChatStore` → `{ messages, threads, joinRoom(roomId), leaveRoom(roomId), send(roomId,text), loadThreads() }` / message `{ id, roomId, senderId, senderRole, text, createdAt }` |
| `assets/tokens.css` | CSS 변수(`--paper/--ink/--brass/--slate-*/--s-*/--r-*` 등). 미정의 대비 `var(--token, #fallback)` 로 작성 |
