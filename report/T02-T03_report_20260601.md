- **작성일**: 2026-06-01

---

## 1. 무엇을 / 왜

임차인이 임대인에게 연락하기 **전에** 하자를 상담(T-02)하고, 그 결과의 **제안 수선비율**과 함께 임대인에게 제보(T-03)하는 흐름을 구현했다. 핵심은 **수선비율 결정 AI**가 ① 임대인 분담 비율(%)과 ② 그 비율에 따른 임대인 연락 필요 여부를 판정하고, 필요 시 임대인에게 **실시간으로 도달**시키는 것이다.

## 2. 어떻게 — 서버

### 프롬프트 — `prompts/repairRate.prompt.ts` (신규)
- `buildRepairChatSystemPrompt(ctx)` — T-02 대화형 상담(민법상 임대인 수선의무 vs 임차인 통상 관리의무 구분, 특약 우선, 참고용 고지).
- `buildRepairAssessmentSystemPrompt(ctx)` — 수선비율 **판정용**. `{"repairRate","needsLandlord","rationale"}` JSON만 출력하도록 강제.
- `REPAIR_DISCLAIMER` 상수.

### 서비스 — `services/repairRate.service.ts` (신규, 지시서가 이번 구현 지정)
- `assessRepairRate(ctx)` — 단일 LLM(`llm.service.chat`) 호출 → 응답에서 JSON 1개 안전 파싱 → `RepairRateAssessment` 반환. **파싱 실패 시 보수적 기본값**(임대인 확인 권장).
- `decideNeedsLandlord(rate, aiNeedsLandlord)` — 결정적 보정: **분담 비율 ≥ 30% 면 무조건 임대인 연락**, 그 외엔 AI 판정 따름. (임계치 상수화)

### 컨트롤러
- `chat.controller.ts` 에 **`repairChat`**(T-02) 추가 — 임차인 임차 건물·특약 맥락 합성 후, **상담 응답 + 수선비율 판정을 병렬(Promise.all)** 수행해 `{ reply, assessment, disclaimer }` 반환.
- `issue.controller.ts` (신규) — **`createIssue`**(T-03): 필수값 검증 → `decideNeedsLandlord` 로 `aiNeedsLandlord` 결정 → Issue 저장 → 연락 필요 시 건물 소유 임대인에게 **`emitToUser(ownerId, 'issue:reported', …)`** 소켓 전달. **`listIssues`**: `mine=true` 임차인 이력 / 그 외 임대인 수신함(연락 필요 건 상단 정렬).
- `routes/issues.ts` — `GET/POST /api/issues` (`requireAuth`).

## 3. 어떻게 — 클라이언트

- `api/issues.ts`, `stores/issues.ts`(Pinia: mine/inbox, report/loadMine/loadInbox, 소켓 수신용 `prependIncoming`).
- **T-02 `views/tenant/DefectChat.vue`** — 분류 선택 + 상담 채팅. AI 응답마다 **수선비율 카드**(분담 비율·근거·연락권장 플래그) 노출, "이 내용으로 제보하기 →" 로 T-03에 `category/description/rate` prefill 전달. 디스클레이머 상시.
- **T-03 `views/tenant/DefectReport.vue`** — 건물·분류·내용·**제안 수선비율 슬라이더(0–100%)** 폼. 제출 → 이력 화면 이동.
- **T-03(수신) `views/landlord/IssueInbox.vue`** — 임대인 수신함. **연락 필요 건 강조**(brass 좌측 바 + 상태 칩, 색만 아닌 dot+텍스트). 소켓 `issue:reported` 구독 시 실시간 갱신.
- `views/tenant/DefectHistory.vue` — 임차인 제보 이력(아카이빙).
