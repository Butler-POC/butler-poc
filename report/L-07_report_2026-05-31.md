# L-07 공실 등록 — 구현 보고서

> 작성일: 2026-05-31 · 작업자: StabAn-NRH
> 관련: L-01(내 건물 등록). 공실은 **건물에 속한 임대 매물**이며, 추후 **A-01(공실 조회)·A-02(건물주 연결)** 의 원천이 된다.

---

## 1. 무엇을 만들었나

임대인이 자기 건물의 **공실(세입자를 구할 빈 집)** 을 등록·조회·수정·삭제하는 기능이다.

설계 판단:
- 공실은 **OCR 문서가 아니라 임대인이 직접 작성하는 매물 정보**다. 따라서 L-01~L-03과 달리 **LLM/비전 없이 순수 CRUD**로 구성했다.
- 공실은 **건물(Building)에 종속되는 1:N**(`Building.vacancies`). 한 건물에 여러 공실 가능.
- 진입점은 **건물 카드**의 "공실 N건 · 관리" 링크 → 건물별 공실 화면.
- **모집중/마감(status) 토글**을 제공해, 계약 완료된 매물을 삭제하지 않고 마감 처리할 수 있다.
- `status`·`dealType` 등은 추후 **중개업자 공실 조회(A-01)** 가 필터링에 쓰도록 설계했다(`@@index([status])`).

---

## 2. 흐름

```
내 건물 목록 → [건물 카드] "공실 N건 · 관리"
   → L-07 화면 (/app/landlord/buildings/:id/vacancies)
      ├ 공실 목록(거래유형·보증금/월세·면적·층·상태)
      │   └ 모집중 ↔ 마감 토글 / 삭제
      └ ＋ 공실 등록 (직접 입력)
       → 등록 → 목록 갱신
```

---

## 3. 데이터 모델 — `Vacancy` (Building 1:N)

| 필드 | 의미 |
|---|---|
| unit | 동/호 |
| dealType | 거래유형 "MONTHLY"(월세) \| "JEONSE"(전세), 기본 MONTHLY |
| deposit / monthlyRent | 보증금 / 월세(원) (전세는 월세 null) |
| maintenanceFee | 관리비(원) |
| areaM2 | 전용면적(㎡) |
| floor | 층(문자열) |
| roomType | 구조(원룸/투룸 등) |
| availableFrom | 입주 가능일 |
| description | 상세 설명(@db.Text) |
| status | "OPEN"(모집중) \| "CLOSED"(마감), 기본 OPEN |
| contact | 문의 연락처 |

- `Building.vacancies Vacancy[]`, `onDelete: Cascade`.
- 인덱스: `buildingId`, `status`(A-01 조회 대비).
- 건물 목록(`GET /buildings`)에 `_count.vacancies` 포함 → 카드에서 공실 수 표시.

---

## 4. API (모두 임대인 전용, 본인 건물 소유 검증)

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/api/buildings/:id/vacancies` | 건물의 공실 목록 |
| POST | `/api/buildings/:id/vacancies` | 공실 등록 |
| PATCH | `/api/buildings/:id/vacancies/:vacancyId` | 공실 수정(상태 토글 등 부분 수정) |
| DELETE | `/api/buildings/:id/vacancies/:vacancyId` | 공실 삭제 |

- 기존 buildings 라우터(`requireRole('LANDLORD')`)에 추가해 역할 가드 공유.
- 모든 작업은 대상 건물의 **소유자(ownerId) 확인** 후 수행(아니면 404).
- PATCH는 **전달된 필드만** 갱신(`'field' in input` 검사)해 상태만 바꾸는 토글을 효율적으로 지원.
